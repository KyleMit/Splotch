# ADR-0043: Magic Brush Reveals the Coloring Page's Colored Twin via Pattern-Fill Ops

**Status:** Active
**Date:** 2026-07

## Context

Every coloring page ships two aligned assets: the black-and-white line art
(`{page}-{tall,wide}.webp`) shown as the overlay, and a flat-colored twin
(`{page}-{tall,wide}.color.webp`, generated by `scripts/gen-coloring-fills.mjs`).
The twin keeps the page's exact outlines and fills the white regions with solid
color, pixel-aligned to the line art. That script was written ahead of any
consumer, explicitly "so a future brush tool can pair each page with its colored
twin and reveal the prefilled colors as a child paints."

This ADR is that consumer: a **magic brush** that reveals the twin's colors
wherever the child strokes, shown in the Actions Panel only while a coloring page
is applied. The requirements that shaped the design:

- **Performance-sensitive.** The drawing hot path is already tuned (ADR-0032/0036);
  the brush must not regress per-move draw cost.
- **Easy undo.** Undo must stay cheap and correct (ADR-0033).
- **Correct interaction ordering.** The eraser must remove magic pixels, and a
  solid stroke drawn *after* a magic stroke must paint over it — the same
  draw-order semantics every other tool obeys.

The obvious first idea — keep a single mask over the whole colored layer and
composite it as a separate layer above the strokes — fights all three: a separate
top layer can't be overridden by later solid strokes, the eraser would have to
punch a second surface, and undo would need bespoke mask snapshots outside the
command log.

## Decision

A magic stroke is an **ordinary op in the existing command log** (ADR-0033),
flagged `magic`, whose paint is a **`CanvasPattern` of the colored twin** instead
of a solid color. The twin is decoded once and rasterized into an offscreen
"sheet" canvas the exact size of the main canvas, positioned with the same
`object-fit: contain` math the overlay `<img>` uses, so a revealed pixel lands
under the line art it belongs to. `renderOp` strokes that pattern for a magic op;
everything else is unchanged.

Because magic ops live in the **same canvas, in draw order**, every hard
requirement falls out of machinery that already exists:

- **Undo** replays them like any op — no new snapshot/mask layer (ADR-0033).
- **Eraser** is `destination-out` on the shared canvas, so it clears revealed
  pixels exactly as it clears solid ones.
- **Override** is just source-over order: a later solid stroke paints on top.
- **Simplification** (ADR-0036) thins them like any path op; `magic` joins the
  per-run style key so magic and solid runs don't merge.

The pattern samples the sheet canvas directly, so the shipping path never reads
the sheet's pixels back — no `getImageData` on load or resize. (An early cut did,
for the rejected `sample` technique below, and it dominated the resize rebuild
cost — dropping it took that rebuild from ~127 ms to ~17 ms.)

### Empirically chosen over two alternatives

All three techniques were prototyped behind a throwaway `setMagicFillMode` seam
and measured with a `perf:magic` harness driving the same stroke battery over a
sheet through `/dev/engine` (2× DPR, keyframing disabled to isolate replay cost).
The numbers below are that comparison; once it settled the decision, the two
rejected paths and the seam/harness were removed — this ADR is their record:

| technique | draw avg (ms/move) | draw max | full rebuild | undo all | segments |
|---|---|---|---|---|---|
| baseline (plain pen) | 0.019 | 0.30 | 0.5 ms | 0.7 ms | 332 |
| **pattern (shipped)** | **0.015** | 3.70 | 16.7 ms | **0.8 ms** | 332 |
| mask (whole-layer composite) | **24.4** | 69.8 | 869 ms | 4505 ms | 332 |
| sample (bake sampled color) | 0.012 | 0.50 | 1.0 ms | 3.0 ms | 726 |

- **`pattern`** — draws at plain-pen cost (the pattern is set once per op, like a
  color) and undoes in ~1 ms; pixel-perfect reveal (the twin's own shading/outline
  detail shows through). The only added cost is a one-off sheet re-rasterize on
  resize/rotation, which already does a full rebuild.
- **`mask`** — the "mask the colored layer" idea, applied per op: stroke an opaque
  shape into a scratch, keep the sheet inside it (`source-in`), blit that. Same
  pixels as `pattern` but a **full-canvas composite every pointermove** →
  ~1600× the per-move cost, blowing the frame budget, and a ~4.5 s teardown to
  undo. Rejected on performance.
- **`sample`** — no magic op at all: sample the sheet color under the pointer and
  store a normal solid stroke. Cheapest draw and free undo, but the per-frame
  sampled colors **fragment the command log** (726 vs 332 segments — different
  colors break simplification runs, so replay costs ~2.2×) and the reveal is
  **flat**, losing the twin's shading and its own outlines. Rejected on fidelity
  (and log bloat).

## Consequences

- **+** Zero new undo/eraser/ordering machinery — the feature rides the command
  log, so the three hard requirements are correct by construction, not by a
  parallel mask system that would have to re-derive them.
- **+** No measurable live-draw regression vs. the plain pen; undo stays ~1 ms.
- **+** Pixel-perfect reveal aligned to the line art, including the twin's shading.
- **−** The rejected alternatives are gone from the code, so re-opening the choice
  (e.g. on a device where pattern raster is the bottleneck) means re-deriving them
  from the numbers above rather than flipping a seam.
- **−** A one-off sheet re-rasterize on resize/rotation (~17 ms in the synthetic
  worst case); acceptable since rotation already forces a full rebuild (ADR-0034).
- **−** Painting in the `contain` letterbox (outside the picture) reveals nothing —
  the pattern is transparent there. Correct (there are no colors to reveal), but a
  stroke in the margin is invisible. The picture dominates the canvas in the
  matching orientation, so this is a thin edge.
- **−** The twin must exist for every page (`bookAssetPaths` now lists the
  `.color.webp` twins so `check-assets` enforces it and native keeps them).
