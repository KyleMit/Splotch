## Source: Code audit

- [ ] **[Performance] AiDial's rAF loop is orphaned when the error branch unmounts it** — File(s): `web/src/lib/components/AiDial.svelte`, `web/src/lib/components/AiImageResult.svelte`
  `AiDial` starts a self-rescheduling `requestAnimationFrame` loop (`loop`, lines 24–46) and relies on `$effect(() => { if (ui.aiError) stopDial(); })` (74–76) to stop it — but none of its effects registers a teardown. When generation fails, `failGeneration` sets `ui.aiError = true` while the result stage stays open, and `AiImageResult`'s `{#if ui.aiError}` branch (line 97) unmounts `<AiDial>` (line 150) before the child's queued `stopDial` effect runs — a destroyed component's pending effects are discarded. With `done` still false the loop reschedules forever (the "waiting" branch never terminates), leaving a permanent per-frame CPU drain, and each retry stacks another orphan loop. Fix: add unmount cleanup — `$effect(() => () => cancelAnimationFrame(rafId))` or `onDestroy(stopDial)`.

- [ ] **[Maintainability] Invalid managed tokens hit `/api/generate-image` with no rate limit — an unthrottled token oracle** — File(s): `web/src/routes/api/generate-image/+server.ts` (lines 52–66)
  The managed-token check (`if (!usingByok && ... !(await isAllowedToken(token))) throw error(403)`) runs *before* `rateLimit(...)`, and the limiter keys are per valid token / per BYOK IP — so a brute-forcer guessing access codes never reaches any limiter: each guess gets an instant 403 plus a free Netlify Blobs read. This violates the repo's own rule (`.claude/rules/server-api.md` / ADR-0014: any unauthenticated oracle must be rate-limited per IP) and is exactly the oracle `/api/verify-access-code` throttles. Fix: apply a per-IP limiter before (or on failure of) the token check, e.g. `rateLimit(\`generate-image-denied:${getClientAddress()}\`)` around the 403 path. Secondary (same handler): `await request.formData()` buffers the full multipart body before auth/throttle/size checks run, so the `MAX_IMAGE_BYTES` guard fires only after the bytes are already in memory — note it if restructuring, but the missing limiter is the actionable fix.

- [ ] **[Maintainability] `clearCanvas()` during an in-flight stroke resurrects wiped ink on replay** — File(s): `web/src/lib/drawing/engine.ts` (lines 1395–1405, 1146–1155)
  `clearCanvas()` pushes a `{ kind: 'clear' }` command and wipes the visible canvas but never commits or discards `activeCommand`. If a pointer is mid-stroke when clear fires (reachable in this multi-touch app: `dragToClear` releases pointers at drag *start*, but a second finger can begin a stroke during the drag and still be down when `onClear` fires), the stroke's pre-clear ops stay in `activeCommand` and `commitActiveCommand()` pushes them *after* the clear on lift. Every rebuild (`paintStateThrough` → undo, resize, rotation, export) then replays clear-then-stroke, repainting ink the user saw erased. Fix: have `clearCanvas()` commit (or split) the open `activeCommand` first — e.g. call `commitActiveCommand()` and re-open a fresh command for still-down pointers — so replay order matches what was rendered.

- [ ] **[Maintainability] Admin console failure paths are silent — errors vanish into unhandled rejections or an invisible flash** — File(s): `web/src/lib/components/admin/AdminConsole.svelte` (lines 65–73, 204–213), `web/src/routes/admin/+page.svelte` (28–40), `web/src/routes/admin/native/+page.svelte` (46–59, 82–100)
  Three compounding gaps: (1) `AdminConsole.run()` is `try { await fn(); } finally { busy = false; }` with no catch, and callers invoke it as a floating promise — any rejection is an unhandled promise rejection with zero UI feedback. (2) The web door's `submit()` does a bare `fetch(action, …)` with no try/catch, so an offline click on Sign in / Add / Remove rejects straight through `run()`. (3) The native door's post-login snapshot fetch (`return applySnapshot(await authedFetch('GET'))`, line 100) sits outside its try/catch. Related: `applySnapshot` reports non-401 failures via `flash`, but `AdminConsole` renders `flash` only in the authed `{:else}` branch — a failed stored-session GET during `onMount` sets a message the user can never see. Fix: catch in `run()` and surface a generic flash/`loginError`; move line 100 inside the try; route unauthed-state errors to `loginError`.

- [ ] **[Maintainability] Token allowlist mutations are bare read-modify-write while `usage.ts` already has the CAS pattern to copy** — File(s): `web/src/lib/server/tokens.ts` (lines 76–82, 111–131), `web/src/lib/server/usage.ts` (51–63)
  `addToken`/`removeToken` do an eventual-consistency `readStore()` then `persist()` — a bare `store.setJSON(KEY, list)` with no etag condition. Two concurrent mutations (web `/admin` form action + native `/api/admin/tokens`, or two admins) lose one write, and a stale replica read followed by `persist` silently drops a token another admin just added — the exact clobber `removeToken`'s own no-op guard comment (126–128) worries about. `recordTokenUsage` solves the identical problem with `getWithMetadata` + `onlyIfMatch`/`onlyIfNew` + retry; port that pattern into `persist`. While there: `removeToken` should also best-effort `getStore('ai-usage').delete(token)` — revoked tokens currently leave their usage blob (keyed by the raw secret, including `lastPrompt`) forever, and re-adding the same token inherits the stale tally.

- [ ] **[Architecture] Engine singleton state survives component remounts — no reset in `initDrawingCanvas`, no dispose in `teardown()`** — File(s): `web/src/lib/drawing/engine.ts` (lines 149–334, 1266–1355)
  All engine state is module-level (`commandLog`, `baselineCanvas`, `canvasEmpty`, `activeCommand`, `paper*`, `activePointers`, `liveDownIds`, …) but `initDrawingCanvas()` resets only `canvas`, `ctx`, callbacks, `currentColor`, and `renderScale`, and `teardown()` only removes listeners. `DrawingCanvas.svelte` remounts on client-side navigation away from `/` and back (e.g. `/privacy`), and `/dev/engine` mounts the same singleton — the new mount silently inherits the previous command log, baseline raster, empty flag, and stale `liveDownIds` entries (their `pointerup` tracker was unhooked, so `isOrphanPenContact` can permanently misclassify a reused pointer id). Fix: explicit reset in `initDrawingCanvas` (or a documented "persists by design" decision), and have `teardown()` call `releaseAllPointers()` and clear `liveDownIds`. This also creates the seam for unit-testing the replay/regroup logic — `engine.ts` (1527 lines) currently has zero Vitest coverage.

- [ ] **[Performance] CI wastes minutes on every push: no Playwright browser cache, no concurrency groups, and blobs-smoke runs a full `npm ci` for a dependency-free script** — File(s): `.github/workflows/test.yml` (lines 61–62), `.github/workflows/blobs-smoke.yml` (lines 41–48)
  (1) `npx playwright install --with-deps chromium` re-downloads ~100+ MB on every run — add `actions/cache` on `~/.cache/ms-playwright` keyed to the `@playwright/test` version from `package-lock.json` (cache the browser dir; run `install-deps` separately). (2) No workflow declares a `concurrency` group, so rapid pushes to the same branch stack full duplicate runs — add `concurrency: { group: "${{ github.workflow }}-${{ github.ref }}", cancel-in-progress: true }` to `test.yml`. (3) `blobs-smoke.yml` fires on every successful deploy and installs the entire root tree (Capacitor, sharp, Playwright, patch-package postinstall) — but `scripts/blobs-smoke.mjs` imports only node builtins plus `./lib/utils.mjs`/`./lib/smoke.mjs` (also builtin-only); replace the install + npm-run steps with a direct `node scripts/blobs-smoke.mjs`.

- [ ] **[Maintainability] Three scripts break the ADR-0017 cross-platform guarantee** — File(s): `scripts/perf/analyze.mjs` (line 425), `scripts/android-emulator-smoke.mjs` (78–81), `scripts/lib/android.mjs` (11–16)
  (1) `analyze.mjs`'s main-module guard `import.meta.url === \`file://${process.argv[1]}\`` never matches on Windows (`file://C:\...` vs `file:///C:/...`), so `npm run perf:analyze` exits 0 having done nothing — compare via `pathToFileURL(process.argv[1]).href`. (2) `android-emulator-smoke.mjs` spawns bare `gradlew.bat`, the exact form `scripts/gradle.mjs`'s header documents as broken under `NoDefaultCurrentDirectoryInExePath=1` — invoke the wrapper by absolute path (or share a helper from `scripts/lib/android.mjs`). (3) `android.mjs`'s non-Windows SDK fallback is the macOS path (`~/Library/Android/sdk`) with no Linux branch (`~/Android/Sdk`), so ADB/emulator discovery fails on Linux unless `ANDROID_HOME` is set.

- [ ] **[Maintainability] Master-key get-then-create race in web secure storage can permanently destroy the stored API key** — File(s): `web/src/lib/secureStorage.ts` (lines 50–71)
  `getMasterKey` is get-then-create with no memoization: two concurrent first-time callers (e.g. `saveApiKey` + `saveAdminSession`, or two tabs) can both see "no key", generate two different non-extractable keys, and the second `db.put` overwrites the first — any payload encrypted with the losing key becomes permanently undecryptable, and `webLoad`'s catch silently reports "no value". Fix in-tab by caching the promise (`masterKeyPromise ??= …`) so all callers share one get-or-create (also removes the redundant IDB get per save/load); cross-tab, do the create inside a single readwrite transaction that re-checks before putting. Land it with tests — `secureStorage.ts` is the only storage module with no colocated test, and `hydrateApiKey`'s legacy-plaintext migration (`settings.svelte.ts` 229–245) is also unexercised; the module's lazy `getPlugin`/`getDb` thunks are already mock-friendly.

- [ ] **[Maintainability] `localStorage` reads are unguarded while writes are carefully guarded — a throwing localStorage crashes boot** — File(s): `web/src/lib/storage.ts` (lines 71, 87, 115 vs 33–42)
  `safeLocalStorage` exists precisely because storage calls can throw (QuotaExceededError, SecurityError in locked-down/private-mode WebViews), yet every read path calls `localStorage.getItem(key)` bare — and those reads run at module-init time of `settings.svelte.ts`/`strokeWidth.svelte.ts` (inside `$state` initializers), so in a context where touching `localStorage` throws, the whole app fails to boot instead of degrading to defaults. Wrap reads in try/catch returning the fallback, mirroring the write side's once-only warn.

- [ ] **[Performance] Undo keyframe rasters are unbounded in count — worst case ~10 × ~30 MB retained** — File(s): `web/src/lib/drawing/engine.ts` (lines 1173–1189, 244–258, 121–125)
  `maybeKeyframe` allocates a keyframe canvas at `baselineCanvas` size — a `max(w,h)` square at up to 2× DPR (a 1024×1366 iPad → 2732² × 4 B ≈ 30 MB). The comments bound *when* a keyframe fires (outlier commands past 384 segments) but nothing bounds *how many* commands hold one: `MAX_UNDO_STACK_SIZE = 10` pathological scribbles (the exact "finger held down for a minute" case the comment cites) retain ≈ 300 MB — enough to get a WKWebView killed on older iPads. Cheap cap: when creating a new keyframe while an older one exists, fold history through the older keyframed command into the baseline (its keyframe *is* the cumulative state — `foldOldestIntoBaseline` already blits it), trading undo depth for bounded memory. Only the most recent at-or-below keyframe is used by `paintStateThrough` for forward replay.

- [ ] **[Architecture] Platform detection drifts from its single source of truth** — File(s): `web/src/lib/components/parent/SetupInstructions.svelte` (38–40, 57–69, 163–200), `web/src/lib/state/fullscreen.svelte.ts` (28), `web/src/lib/state/install.svelte.ts` (68, 75)
  Two violations of the project's own rules: (1) `SetupInstructions` populates `let native = $state(false)` from runtime `isNative()` on every modal open and forks large markup on it — web-vs-native is a build-time fact (CLAUDE.md: prefer `__IS_CAPACITOR__`; `AboutTab.svelte` 22–31 documents why runtime `isNative()` can even mis-read). Replace with `const native = __IS_CAPACITOR__` and keep only the ios/android re-detection. (2) `/android/i.test(navigator.userAgent || '')` appears verbatim in `install.svelte.ts` (twice) and `fullscreen.svelte.ts` — while `install.svelte.ts` declares itself the single source of "what kind of device is this". Move `isAndroidBrowser()`/`isIosDevice()` into `web/src/lib/platform.ts` (the established platform-detection home) and consume from both state modules.

- [ ] **[Maintainability] The requestIdleCallback-with-setTimeout-fallback pattern is hand-rolled five times** — File(s): `web/src/lib/components/DrawingCanvas.svelte` (110–130, 147–156), `web/src/lib/components/ColoringBook.svelte` (33–40), `web/src/routes/+page.svelte` (52–57), `web/src/lib/drawing/engine.ts` (1324–1330)
  Each site re-implements `typeof requestIdleCallback === 'function' ? requestIdleCallback(fn) : setTimeout(fn, 200)`, the `as unknown as number` cast, the matching cancel bookkeeping, and the same "iOS lacks requestIdleCallback" WHY-comment — DrawingCanvas alone carries two copies with different cancel plumbing, and the copies are already drifting. Extract a `scheduleIdle(fn): () => void` helper (e.g. `web/src/lib/idle.ts`) returning a cancel function; all five sites collapse to one line and the Safari-floor rationale lives in one place.

  Two related token gaps: (1) the muted corner-button idiom (transparent 44–48px button, `opacity: .4 → .7 → 1` ladder, the three-state `invert()` icon filter, `touch-action: manipulation`) is triplicated across ActionsPanel/FullscreenToggle/ParentHelpButton (~35 lines each; ActionsPanel's comment even says it "mirrors" the other) — add `.corner-button`/`.corner-button-icon` to `app.css` beside the existing shared `.modal-close-btn`. - [ ] **[Maintainability] Rate-limit 429s reach parents as the wrong message, and `verify-access-code` omits the shared `error` field** — File(s): `web/src/lib/components/parent/AiKeyManager.svelte` (116–123), `web/src/lib/drawing/aiImage.ts` (82–90), `web/src/routes/api/verify-access-code/+server.ts` (21, 24)
  `http.ts` documents the 429 contract as "clients can surface the same `error` field", but the access-code branch of `AiKeyManager` ignores `data.error` — a throttled parent with a *valid* code is told the code is invalid (the API-key branch correctly falls back to `data.error`); `aiImage.ts` treats any non-422 failure as generic "Something went wrong", ignoring the 429 body and `Retry-After`; and `verify-access-code` returns bare `json({ ok: false })` with no `error` field, drifting from its twin `verify-key`. Fix: prefer `data.error` in both client branches (optionally special-case 429 in `aiImage.ts` with "wait a moment" guidance) and add an `error` field to `verify-access-code` failures.
