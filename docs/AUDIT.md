- [ ] **[Maintainability] Token allowlist mutations are bare read-modify-write while `usage.ts` already has the CAS pattern to copy** ‚Äî File(s): `web/src/lib/server/tokens.ts` (lines 76‚Äì82, 111‚Äì131), `web/src/lib/server/usage.ts` (51‚Äì63)
  `addToken`/`removeToken` do `readStore()` then `persist()` ‚Äî a bare `store.setJSON(KEY, list)` with no etag condition (76‚Äì82). Two concurrent mutations (web `/admin` form action + native `/api/admin/tokens`, or two admins) lose one write ‚Äî the exact clobber `removeToken`'s own no-op guard comment (126‚Äì128) worries about. `recordTokenUsage` solves the identical problem with `getWithMetadata` + `onlyIfMatch`/`onlyIfNew` + a 3-attempt retry (usage.ts:51‚Äì63; `tokens.ts:62` already uses `onlyIfNew` for the seed). Porting notes: (a) the etag must be threaded through `readStore`'s return shape (its other callers can ignore it); (b) the retry loop must wrap the *whole* read-modify (re-running the dup-check/filter), i.e. restructure `addToken`/`removeToken`, not just `persist`; (c) do **not** copy usage.ts's concede-after-retries behavior ‚Äî admin mutations must surface exhausted retries as `{ ok: false, error }` so the `/admin` form action and `/api/admin/tokens` POST report the failure instead of trading a silent clobber for a silent drop (ADR-0025 eventual consistency means retries can exhaust under replica lag). While there: `removeToken` should best-effort `getStore('ai-usage').delete(token)` (must not fail the remove) ‚Äî revoked tokens currently leave a usage blob keyed by the raw secret forever, and re-adding the same token inherits the stale tally. (`lastPrompt` in that blob is server-built from a style enum, not child content, so it's residue hygiene rather than a privacy leak.)

- [ ] **[Architecture] Engine singleton keeps rasters and stale pointer state alive across unmounts ‚Äî make the lifecycle explicit** ‚Äî File(s): `web/src/lib/drawing/engine.ts` (state at 156‚Äì333, `initDrawingCanvas` 1266‚Äì1355, `teardown` 1333‚Äì1353)
  The module-singleton is intentional (ADR-0004), and inheriting `commandLog` across `/` ‚Üí `/privacy` ‚Üí `/` means the child's drawing survives navigation ‚Äî plausibly desirable, and `canvasState` persists in step so `canUndo`/`canvasEmpty` don't desync. But the lifecycle is implicit and leaky: `teardown()` only removes listeners, so the ~30 MB baseline square plus any keyframes stay retained the whole time the user sits on `/privacy`; `initDrawingCanvas()` resets only canvas/ctx/magic-brush wiring/callbacks/`currentColor`/`renderScale`; and stale `activePointers`/`liveDownIds` entries survive teardown ‚Äî a stale `isDrawing: true` entry could let hover moves draw after remount for a reused pointer id (the `liveDownIds` staleness self-heals on the next tracked `pointerup`, so it's a one-off glitch, not permanent). `/dev/engine` mounts the same singleton and inherits a dirty log when reached by client-side nav. Fix: have `teardown()` call `releaseAllPointers()` and clear `liveDownIds`; decide explicitly whether drawing persistence across navigation is wanted and document it (ADR-0004 or code comment) ‚Äî if not, reset the log/rasters in `initDrawingCanvas`. Either way this creates the seam for unit-testing the replay/regroup logic ‚Äî `engine.ts` (~1530 lines) has zero Vitest coverage.

- [ ] **[Performance] CI wastes minutes on every push: no Playwright browser cache, no concurrency groups, and blobs-smoke runs a full `npm ci` for a dependency-free script** ‚Äî File(s): `.github/workflows/test.yml` (lines 61‚Äì62), `.github/workflows/blobs-smoke.yml` (lines 41‚Äì48)
  (1) `npx playwright install --with-deps chromium` re-downloads ~100+ MB on every run ‚Äî no workflow has any `actions/cache` step (setup-node's `cache: npm` caches only the npm download cache). (2) No workflow declares a `concurrency` group, so rapid pushes stack full duplicate runs ‚Äî add `concurrency: { group: "${{ github.workflow }}-${{ github.ref }}", cancel-in-progress: true }` to `test.yml` only (the tag-triggered `android-deploy.yml`/`ios-deploy.yml` are releases where `cancel-in-progress` would be wrong). (3) `blobs-smoke.yml` fires on every successful deploy and installs the entire root tree (Capacitor, sharp, Playwright, `patch-package` postinstall) ‚Äî but `scripts/blobs-smoke.mjs` imports only node builtins plus `./lib/utils.mjs`/`./lib/smoke.mjs` (both builtin-only, verified transitively); replace the `npm ci` + npm-run steps (and the now-pointless `cache: npm` line) with a direct `node scripts/blobs-smoke.mjs`.

- [ ] **[Maintainability] Master-key get-then-create race in web secure storage can permanently destroy the stored API key** ‚Äî File(s): `web/src/lib/secureStorage.ts` (lines 50‚Äì71), `web/src/lib/state/settings.svelte.ts` (229‚Äì245)
  `getMasterKey` (50‚Äì59) is get-then-create with no memoization: concurrent first-time callers can both see "no key", generate two different non-extractable keys, and the second `db.put` overwrites the first ‚Äî any payload encrypted with the losing key becomes permanently undecryptable, and `webLoad`'s catch (81‚Äì83) silently reports "no value". The credible trigger is **two tabs racing the legacy-plaintext migration at boot**: `hydrateApiKey` (settings.svelte.ts:229‚Äì245) runs unconditionally on mount, and two simultaneously-restored tabs holding a legacy plaintext key both call `saveApiKey` concurrently, then both `removeKey` the plaintext copy (line 239) ‚Äî destroying the key with no fallback. (In-tab concurrency is implausible: `saveAdminSession` is native-only/Keychain, and the key-save flow is a single awaited button action; load paths only call `getMasterKey` when a ciphertext record already exists.) Fix in-tab by caching the promise (`masterKeyPromise ??= ‚Ä¶`, cleared on rejection so one transient IDB failure doesn't poison future saves); cross-tab, do the get‚Üíre-check‚Üíput inside a single explicit `db.transaction(STORE, 'readwrite')` (the idb shortcut `db.get`/`db.put` each open their own transaction) with the key generated *before* the transaction ‚Äî an IDB transaction auto-commits when control returns to the event loop, so you can't `await crypto.subtle.generateKey` inside it. Land it with tests ‚Äî there is no `secureStorage.test.ts` and `hydrateApiKey`'s migration is unexercised; the module's lazy `getPlugin`/`getDb` thunks are already mock-friendly.

- [ ] **[Maintainability] `localStorage` reads are unguarded while writes are carefully guarded ‚Äî a throwing localStorage crashes boot** ‚Äî File(s): `web/src/lib/storage.ts` (lines 71, 87, 115 vs 33‚Äì42)
  `safeLocalStorage` exists precisely because storage calls can throw (its own comment names SecurityError in locked-down/private-mode WebViews), yet every read path (`readBool`:71, `readString`:87, `readInt`:115) touches `localStorage` bare ‚Äî and those reads run at module-init time inside `$state` initializers (`settings.svelte.ts:145‚Äì156`, `strokeWidth.svelte.ts:24‚Äì28`), so in a context where accessing the `localStorage` global throws (Chrome "block all cookies", sandboxed iframes, `dom.storage.enabled=false`), client module evaluation fails and hydration dies ‚Äî painted-but-dead UI, since SSR is unaffected. The repo already treats this environment as supported: the inline boot script in `app.html:62‚Äì93` wraps its own `localStorage` reads in try/catch ("private-mode / disabled storage ‚Äî fall back to CSS defaults") ‚Äî that's the degrade model to copy. Fix: a `safeRead` mirroring `safeLocalStorage` that wraps the *whole* read expression (the throw is on the global access, not just `getItem`) and returns the fallback; consider adding a localStorage row to `docs/COMPATIBILITY.md`'s risk register.

- [ ] **[Performance] Undo keyframe rasters are bounded only by the 10-command stack cap ‚Äî worst case ~300 MB retained** ‚Äî File(s): `web/src/lib/drawing/engine.ts` (lines 1173‚Äì1189, 244‚Äì258, 1216‚Äì1235)
  `maybeKeyframe` (1173‚Äì1189) allocates a keyframe canvas at `baselineCanvas` size ‚Äî a `max(w,h)` square at up to 2√ó DPR (a 1024√ó1366 iPad ‚Üí 2732¬≤ √ó 4 B ‚âà 30 MB). The comments bound *when* a keyframe fires (outlier commands past 384 post-simplification segments) but nothing bounds *how many* of the ‚â§ `MAX_UNDO_STACK_SIZE = 10` retained commands hold one: 10 pathological scribbles (the exact "finger held down for a minute" case the comment cites) retain ‚âà 300 MB on top of the baseline + visible canvas + magic sheet ‚Äî enough to get a WKWebView killed on older iPads. Real sessions peak ~140 segments per the comments, so this is worst-case hardening, not a common-path leak. Cheap cap: when creating a new keyframe while an older one exists, fold history *through* the older keyframed command into the baseline ‚Äî repeatedly shift via `foldOldestIntoBaseline` (1196‚Äì1207, which already blits a keyframed command wholesale) until that command is folded. Don't just drop the older raster: it isn't dead ‚Äî undo can pop the newer keyframed command, after which `paintStateThrough` (1216‚Äì1235) would start from the older one. Folding trades undo depth for bounded memory, which is the accepted cost.
