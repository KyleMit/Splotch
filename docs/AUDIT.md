## Source: Code audit

- [ ] **[Maintainability] Admin console failure paths are silent — network rejections vanish, and `flash` can be set where it's never rendered** — File(s): `web/src/lib/components/admin/AdminConsole.svelte` (lines 65–73, 204–213), `web/src/routes/admin/+page.svelte` (28–40), `web/src/routes/admin/native/+page.svelte` (46–59, 82–100)
  Three compounding gaps: (1) `AdminConsole.run()` is `try { await fn(); } finally { busy = false; }` with no catch (65–73), and every call site (75–80, 82–87, 168, 290, 356–359) invokes it as a floating promise — any rejection is an unhandled promise rejection with zero UI feedback. Nuance: server-side failures returned as ActionData *do* surface via the `form` prop (`admin/+page.svelte:42–48`); the silent class is specifically network failures / thrown `deserialize`. (2) The web door's `submit()` does a bare `fetch(action, …)` (31–35) with no try/catch, so an offline click on Sign in / Add / Remove rejects straight through `run()`. (3) The native door's post-login snapshot fetch (`return applySnapshot(await authedFetch('GET'))`, line 100) sits outside its try/catch (82–91) — and by then the session is already saved to secure storage (line 98), so on rejection the user is stuck on the login form with no message despite holding a valid session (a reload recovers via `onMount`). Related: `applySnapshot` reports non-401 failures via `flash` (52–54), but `AdminConsole` renders `flash` only in the authed `{:else}` branch (204–213) while the unauthed branch renders only `loginError` (178–180) — so both the stored-session GET during `onMount` *and* a non-401 failure on the login path set a message the user can never see. Fix: catch in `run()` and surface a generic flash/`loginError`; move line 100 inside the try; route unauthed-state errors to `loginError`.

- [ ] **[Maintainability] Token allowlist mutations are bare read-modify-write while `usage.ts` already has the CAS pattern to copy** — File(s): `web/src/lib/server/tokens.ts` (lines 76–82, 111–131), `web/src/lib/server/usage.ts` (51–63)
  `addToken`/`removeToken` do `readStore()` then `persist()` — a bare `store.setJSON(KEY, list)` with no etag condition (76–82). Two concurrent mutations (web `/admin` form action + native `/api/admin/tokens`, or two admins) lose one write — the exact clobber `removeToken`'s own no-op guard comment (126–128) worries about. `recordTokenUsage` solves the identical problem with `getWithMetadata` + `onlyIfMatch`/`onlyIfNew` + a 3-attempt retry (usage.ts:51–63; `tokens.ts:62` already uses `onlyIfNew` for the seed). Porting notes: (a) the etag must be threaded through `readStore`'s return shape (its other callers can ignore it); (b) the retry loop must wrap the *whole* read-modify (re-running the dup-check/filter), i.e. restructure `addToken`/`removeToken`, not just `persist`; (c) do **not** copy usage.ts's concede-after-retries behavior — admin mutations must surface exhausted retries as `{ ok: false, error }` so the `/admin` form action and `/api/admin/tokens` POST report the failure instead of trading a silent clobber for a silent drop (ADR-0025 eventual consistency means retries can exhaust under replica lag). While there: `removeToken` should best-effort `getStore('ai-usage').delete(token)` (must not fail the remove) — revoked tokens currently leave a usage blob keyed by the raw secret forever, and re-adding the same token inherits the stale tally. (`lastPrompt` in that blob is server-built from a style enum, not child content, so it's residue hygiene rather than a privacy leak.)

- [ ] **[Architecture] Engine singleton keeps rasters and stale pointer state alive across unmounts — make the lifecycle explicit** — File(s): `web/src/lib/drawing/engine.ts` (state at 156–333, `initDrawingCanvas` 1266–1355, `teardown` 1333–1353)
  The module-singleton is intentional (ADR-0004), and inheriting `commandLog` across `/` → `/privacy` → `/` means the child's drawing survives navigation — plausibly desirable, and `canvasState` persists in step so `canUndo`/`canvasEmpty` don't desync. But the lifecycle is implicit and leaky: `teardown()` only removes listeners, so the ~30 MB baseline square plus any keyframes stay retained the whole time the user sits on `/privacy`; `initDrawingCanvas()` resets only canvas/ctx/magic-brush wiring/callbacks/`currentColor`/`renderScale`; and stale `activePointers`/`liveDownIds` entries survive teardown — a stale `isDrawing: true` entry could let hover moves draw after remount for a reused pointer id (the `liveDownIds` staleness self-heals on the next tracked `pointerup`, so it's a one-off glitch, not permanent). `/dev/engine` mounts the same singleton and inherits a dirty log when reached by client-side nav. Fix: have `teardown()` call `releaseAllPointers()` and clear `liveDownIds`; decide explicitly whether drawing persistence across navigation is wanted and document it (ADR-0004 or code comment) — if not, reset the log/rasters in `initDrawingCanvas`. Either way this creates the seam for unit-testing the replay/regroup logic — `engine.ts` (~1530 lines) has zero Vitest coverage.

- [ ] **[Performance] CI wastes minutes on every push: no Playwright browser cache, no concurrency groups, and blobs-smoke runs a full `npm ci` for a dependency-free script** — File(s): `.github/workflows/test.yml` (lines 61–62), `.github/workflows/blobs-smoke.yml` (lines 41–48)
  (1) `npx playwright install --with-deps chromium` re-downloads ~100+ MB on every run — no workflow has any `actions/cache` step (setup-node's `cache: npm` caches only the npm download cache). Add `actions/cache` on `~/.cache/ms-playwright` keyed to the `@playwright/test` version from `package-lock.json`; on cache hit still run `npx playwright install-deps chromium` (the apt packages aren't cacheable). (2) No workflow declares a `concurrency` group, so rapid pushes stack full duplicate runs — add `concurrency: { group: "${{ github.workflow }}-${{ github.ref }}", cancel-in-progress: true }` to `test.yml` only (the tag-triggered `android-deploy.yml`/`ios-deploy.yml` are releases where `cancel-in-progress` would be wrong). (3) `blobs-smoke.yml` fires on every successful deploy and installs the entire root tree (Capacitor, sharp, Playwright, `patch-package` postinstall) — but `scripts/blobs-smoke.mjs` imports only node builtins plus `./lib/utils.mjs`/`./lib/smoke.mjs` (both builtin-only, verified transitively); replace the `npm ci` + npm-run steps (and the now-pointless `cache: npm` line) with a direct `node scripts/blobs-smoke.mjs`.

- [ ] **[Maintainability] Three scripts break the ADR-0017 cross-platform guarantee** — File(s): `scripts/perf/analyze.mjs` (line 425), `scripts/android-emulator-smoke.mjs` (78–81), `scripts/lib/android.mjs` (11–16)
  (1) `analyze.mjs`'s main-module guard `import.meta.url === \`file://${process.argv[1]}\`` never matches on Windows (`file://C:\...` vs `file:///C:/...`), so `npm run perf:analyze` exits 0 having done nothing — compare via `pathToFileURL(process.argv[1]).href` (precedent: `scripts/redteam-run.mjs:21`). The guard is load-bearing, not deletable: `analyze.mjs` is imported as a library by `perf/session.mjs`, `perf/undo-scenarios.mjs`, and `perf/replay-scenario.mjs`. (2) `android-emulator-smoke.mjs` spawns bare `gradlew.bat` (78–81), the exact form `scripts/gradle.mjs`'s header documents as broken under `NoDefaultCurrentDirectoryInExePath=1` — resolve the wrapper to an absolute path like gradle.mjs:17–19 does, keeping `sh()` (ADR-0017 explicitly blesses it here) and quoting the path in the command string (cf. the `"${maestroPath()}"` pattern at line 82). (3) `android.mjs`'s non-Windows SDK fallback is only the macOS path (`~/Library/Android/sdk`) — no Linux branch (`~/Android/Sdk`) and no `ANDROID_SDK_ROOT` fallback, so ADB/emulator discovery fails on Linux unless `ANDROID_HOME` is set; blast radius is local Linux dev only (CI presets `ANDROID_HOME`). Note: ADR-0017's literal text guarantees "Windows and macOS" while the root and scripts CLAUDE.md say "macOS, Linux, and Windows" — fixing (3) should also update android.mjs's header comment and ideally the ADR wording.

- [ ] **[Maintainability] Master-key get-then-create race in web secure storage can permanently destroy the stored API key** — File(s): `web/src/lib/secureStorage.ts` (lines 50–71), `web/src/lib/state/settings.svelte.ts` (229–245)
  `getMasterKey` (50–59) is get-then-create with no memoization: concurrent first-time callers can both see "no key", generate two different non-extractable keys, and the second `db.put` overwrites the first — any payload encrypted with the losing key becomes permanently undecryptable, and `webLoad`'s catch (81–83) silently reports "no value". The credible trigger is **two tabs racing the legacy-plaintext migration at boot**: `hydrateApiKey` (settings.svelte.ts:229–245) runs unconditionally on mount, and two simultaneously-restored tabs holding a legacy plaintext key both call `saveApiKey` concurrently, then both `removeKey` the plaintext copy (line 239) — destroying the key with no fallback. (In-tab concurrency is implausible: `saveAdminSession` is native-only/Keychain, and the key-save flow is a single awaited button action; load paths only call `getMasterKey` when a ciphertext record already exists.) Fix in-tab by caching the promise (`masterKeyPromise ??= …`, cleared on rejection so one transient IDB failure doesn't poison future saves); cross-tab, do the get→re-check→put inside a single explicit `db.transaction(STORE, 'readwrite')` (the idb shortcut `db.get`/`db.put` each open their own transaction) with the key generated *before* the transaction — an IDB transaction auto-commits when control returns to the event loop, so you can't `await crypto.subtle.generateKey` inside it. Land it with tests — there is no `secureStorage.test.ts` and `hydrateApiKey`'s migration is unexercised; the module's lazy `getPlugin`/`getDb` thunks are already mock-friendly.

- [ ] **[Maintainability] `localStorage` reads are unguarded while writes are carefully guarded — a throwing localStorage crashes boot** — File(s): `web/src/lib/storage.ts` (lines 71, 87, 115 vs 33–42)
  `safeLocalStorage` exists precisely because storage calls can throw (its own comment names SecurityError in locked-down/private-mode WebViews), yet every read path (`readBool`:71, `readString`:87, `readInt`:115) touches `localStorage` bare — and those reads run at module-init time inside `$state` initializers (`settings.svelte.ts:145–156`, `strokeWidth.svelte.ts:24–28`), so in a context where accessing the `localStorage` global throws (Chrome "block all cookies", sandboxed iframes, `dom.storage.enabled=false`), client module evaluation fails and hydration dies — painted-but-dead UI, since SSR is unaffected. The repo already treats this environment as supported: the inline boot script in `app.html:62–93` wraps its own `localStorage` reads in try/catch ("private-mode / disabled storage — fall back to CSS defaults") — that's the degrade model to copy. Fix: a `safeRead` mirroring `safeLocalStorage` that wraps the *whole* read expression (the throw is on the global access, not just `getItem`) and returns the fallback; consider adding a localStorage row to `docs/COMPATIBILITY.md`'s risk register.

- [ ] **[Performance] Undo keyframe rasters are bounded only by the 10-command stack cap — worst case ~300 MB retained** — File(s): `web/src/lib/drawing/engine.ts` (lines 1173–1189, 244–258, 1216–1235)
  `maybeKeyframe` (1173–1189) allocates a keyframe canvas at `baselineCanvas` size — a `max(w,h)` square at up to 2× DPR (a 1024×1366 iPad → 2732² × 4 B ≈ 30 MB). The comments bound *when* a keyframe fires (outlier commands past 384 post-simplification segments) but nothing bounds *how many* of the ≤ `MAX_UNDO_STACK_SIZE = 10` retained commands hold one: 10 pathological scribbles (the exact "finger held down for a minute" case the comment cites) retain ≈ 300 MB on top of the baseline + visible canvas + magic sheet — enough to get a WKWebView killed on older iPads. Real sessions peak ~140 segments per the comments, so this is worst-case hardening, not a common-path leak. Cheap cap: when creating a new keyframe while an older one exists, fold history *through* the older keyframed command into the baseline — repeatedly shift via `foldOldestIntoBaseline` (1196–1207, which already blits a keyframed command wholesale) until that command is folded. Don't just drop the older raster: it isn't dead — undo can pop the newer keyframed command, after which `paintStateThrough` (1216–1235) would start from the older one. Folding trades undo depth for bounded memory, which is the accepted cost.

- [ ] **[Architecture] Platform detection drifts from its single source of truth** — File(s): `web/src/lib/components/parent/SetupInstructions.svelte` (38, 57–84, 163–201), `web/src/lib/state/fullscreen.svelte.ts` (28), `web/src/lib/state/install.svelte.ts` (68, 75)
  Hygiene rather than a live bug (web builds agree on both signals, and line 69 already mixes in the compile-time flag), but two drifts from the project's own rules: (1) `SetupInstructions` populates `let native = $state(false)` from runtime `isNative()` on every modal open and forks large markup on it — web-vs-native is a build-time fact (`web/src/CLAUDE.md`: prefer `__IS_CAPACITOR__`; `AboutTab.svelte:22–31` documents how runtime `isNative()` can mis-read). Fix gotcha: the sanctioned form is `const native = __IS_CAPACITOR__ && isNative()`, *not* the bare flag — `web/vitest.config.ts:15–18` deliberately defines `__IS_CAPACITOR__: true` in unit tests so runtime `isNative()` mocks stay in control. Keep `getPlatform()` runtime — one `CAPACITOR=true` bundle ships in both the iPhone and Android binaries, so ios-vs-android is a per-device fact. (2) `/android/i.test(navigator.userAgent || '')` appears verbatim in `install.svelte.ts` (twice) and `fullscreen.svelte.ts` — while `install.svelte.ts:64–65` declares itself the single source of "what kind of device is this". Move `isAndroidBrowser()`/`isIosDevice()` into `web/src/lib/platform.ts` (the established platform-detection home, currently zero UA sniffing) and consume from both state modules.

- [ ] **[Maintainability] The requestIdleCallback-with-setTimeout-fallback pattern is hand-rolled five times (three exact copies, two loose variants)** — File(s): `web/src/lib/components/DrawingCanvas.svelte` (112–129, 147–156), `web/src/lib/components/ColoringBook.svelte` (33–40), `web/src/routes/+page.svelte` (52–56), `web/src/lib/drawing/engine.ts` (1325–1330)
  Three sites are identical copies of `typeof requestIdleCallback === 'function' ? requestIdleCallback(fn) : setTimeout(fn, 200)` with the `as unknown as number` cast, cancel bookkeeping, and the same iOS-lacks-requestIdleCallback WHY comment (DrawingCanvas ×2 with different cancel plumbing, ColoringBook ×1); two more are loose variants that are already drifting: `+page.svelte`'s `schedule` stores no handle and relies on a `stopped` flag, and `engine.ts` uses `'requestIdleCallback' in window` with a **0 ms** fallback. Extract a `scheduleIdle(fn): () => void` helper (e.g. `web/src/lib/idle.ts`) returning a cancel function — it drops in cleanly for the three exact copies. Adoption caveats for the variants: `+page.svelte`'s `stopped` flag also guards an async `import().then` continuation a cancel function can't reach, so that flag stays even after adoption; and moving `engine.ts` to a shared 200 ms fallback is a small behavior change (paper-texture warm delayed on Safari) — accept it deliberately or parameterize the delay. No site passes an `options`/`{timeout}` arg, so the signature can stay minimal.

- [ ] **[Maintainability] Rate-limit 429s reach parents as the wrong message, and `verify-access-code` omits the shared `error` field** — File(s): `web/src/lib/components/parent/AiKeyManager.svelte` (112–122), `web/src/lib/drawing/aiImage.ts` (82–101), `web/src/routes/api/verify-access-code/+server.ts` (21, 24)
  `http.ts:17–21` documents the 429 contract as "clients can surface the same `error` field", but: (1) the access-code branch of `AiKeyManager` ignores `data.error` (121–122) — a throttled parent with a *valid* code is told the code is invalid, while the API-key branch correctly falls back (`data.error || …`, 112–113); fix is one-line symmetry, and this is a parent-facing panel where "Too many attempts, wait Ns" is appropriate.
