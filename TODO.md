# Repository Health TODO

Instructions for the AI agent: address these recommendations one at a time. For each item, inspect the referenced code, make the smallest coherent improvement, run the relevant checks/tests, and then remove that completed recommendation from this file in the same change.

- Harden canvas resize preservation in `src/lib/drawing/engine.ts`. The virtual canvas is created once at `Math.max(rect.width, rect.height) * 2` and never grows. If the viewport grows beyond that initial size (e.g., device rotates to a larger dimension on second mount), `resizeCanvas()` will silently clip drawing data. Fix: inside `resizeCanvas()`, after the `if (!virtualCanvas)` block, check whether the existing virtual canvas is too small for the current `rect` (`rect.width * 2 > virtualCanvas.width || rect.height * 2 > virtualCanvas.height`), and if so create a larger replacement, copy the old pixels in, and replace `virtualCanvas`/`virtualCtx`.

- Make undo memory usage explicit in `src/lib/drawing/engine.ts`. Each undo snapshot is a full canvas copy: at 1024×768 (RGBA = 4 bytes) that's ~3 MB per snapshot, or ~30 MB for the stack of 10. On a high-DPI tablet where the virtual canvas is 2× the display size, snapshots can reach 50–100 MB. Add a comment near `MAX_UNDO_STACK_SIZE` documenting this budget so future maintainers can make an informed decision. Consider also whether `MAX_UNDO_STACK_SIZE` should be capped lower for very large canvases, or whether compressed `ImageBitmap`/blob snapshots are worth exploring.

- Reduce main-thread work after erasing in `src/lib/drawing/engine.ts`. `scanCanvasIsEmpty()` reads the full canvas pixel data on every erase-end. `willReadFrequently: true` on the context already avoids the worst case (GPU→CPU texture readback), but on a large canvas the full `getImageData` loop is still non-trivial. A simpler win: defer the scan with `requestIdleCallback` (with a fallback `setTimeout(0)`) so it runs after the gesture frame. A dirty-state approach (track whether any non-erase strokes landed since the last clear) would eliminate the scan for most users but requires more care.

The download-link pattern (`createElement('a')`, set `href`/`download`, `click()`, `remove()`) is also repeated in both files. Extract `timestamp()` and a `triggerDownload(url, filename)` helper into `screenshot.ts` (or a small shared utility) and import from there.
